Google Fact Check Experimental Pipeline

Goal: establish a fully functional and reproducible experimental pipeline.

reproducible no paid X dependency structured artifacts presentation ready

The problem

Original route

  • relied on X/Twitter-style hydration or API access
  • costs scale badly for student work
  • reproducibility suffers when the dataset depends on paid or unstable access

Project response

  • pivot to Google Fact Check Tools + Data Commons
  • keep the base pipeline public and repeatable
  • separate deterministic baseline from optional API-backed model runs

Chosen data sources

  • Google Fact Check Explorer for exploratory sampling and topic inspection
  • Fact Check Tools API for structured claim/query access
  • Data Commons ClaimReview feed for repeatable daily collection
  • Data Commons research dataset for historical structured ClaimReview markup

This is a cleaner foundation for reproducibility than a paid X ingestion path.

Metadata available

claim text claimant claim date review URL review title review publisher review date textual rating language code appearance URLs

Important caveat: the release gives structured ClaimReview metadata, not the full article body. If later experiments need richer evidence, article-body enrichment becomes a separate second stage.

Pipeline architecture

  1. download the public daily feed
  2. normalize each record into a stable JSON schema
  3. build prompt-ready records
  4. run either a deterministic heuristic baseline or an optional OpenAI-compatible model
  5. save predictions + quality report as explicit artifacts
raw feed -> normalized records -> prompts -> predictions -> quality report

Why this is reproducible

  • CLI-first pipeline, not notebook-first
  • deterministic default baseline works without API keys
  • stable IDs and explicit output directories
  • JSON / JSONL / CSV artifacts saved at every stage
  • optional API mode kept separate from the baseline run
python3 src/run_experiment.py --limit 100 --mode heuristic --output-dir artifacts/pilot_run

Known limits

  • the base Google/Data Commons path does not include full article text
  • category labels are stronger in the Explorer UI than in the raw feed
  • the signed-in Chrome session could not be attached from this session, so the presentation is local HTML instead of NotebookLM / Google Slides
  • the Python CLI was authored and read back successfully, but local execution from this session was still blocked by the environment's exec approval state

Next steps

  • run a 100–200 item deterministic pilot
  • inspect missingness, duplicates, language spread, and rating distribution
  • optionally add article-body enrichment
  • optionally compare heuristic baseline vs API-backed model outputs

Bottom line: the project is now structured as a repeatable end-to-end workflow, not just a one-off data scrape.
